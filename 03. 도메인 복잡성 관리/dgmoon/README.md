# ch.03 도메인 복잡성 관리

## 일관성 없는 모델

- 마케팅과 영업부서 간 "리드"라는 용어의 모호성을 해결하기 위한 방법
    - 도메인 주도 설계 패턴 중 하나는 "바운디드 컨텍스트"

## 바운디드 컨텍스트란 무엇인가

- 유비쿼터스 언어를 여러 개의 작은 언어로 나누고 각 언어를 적용할 수 있는 명시적인 바운디드 컨텍스트에 할당

### 모델 경계

- 모델은 경계 없이 존재할 수 없고, 모델의 경계를 정의하는 것은 모델링 프로세스의 본질적인 부분

### 정제된 유비쿼터스 언어

- 유비쿼터스 언어는 바운디드 컨텍스트 경계 안에서만 보편적으로 적용됨

### 바운디드 컨텍스트의 범위

- 바운디드 컨텍스트의 크기에 대한 결정은 문제 도메인이 무엇이냐에 따라 달라지며
- 때로는, 넓은 경계를 사용하는 것이 더 명확하고, 어떤 경우에는 분해하는 것이 더 합리적

## 바운디드 컨텍스트 대 하위 도메인

### 하위 도메인

- 소프트웨어 엔지니어는 요구사항을 정의하지 않음
- 소프트웨어 엔지니어는 하위 도메인을 식별하기 위해 비즈니스 도메인을 분석

### 바운디드 컨텍스트

- 반면, 바운디드 컨텍스트는 소프트웨어 엔지니어에 의해 설계됨

### 하위 도메인과 바운디드 컨텍스트 사이의 상호 작용

- 모놀리식 바운디드 컨텍스트: 비현실적. 소규모 시스템에서 효과적
- 유비쿼터스 언어의 일관성에 따라 나눈 바운디드 컨텍스트: 도메인 전문가의 멘탈 모델을 따라 시스템을 바운디드 컨텍스트로 분해 가능
- 하위 도메인 경계에 맞춰 나눈 바운디드 컨텍스트: 모델이 여전히 크고 유지보수하기 어려운 경우 더 작은 바운디드 컨텍스트로 분해 가능

## 경계

- 설계는 본질적으로 경계에 대한 것

### 물리적 경계

바운디드 컨텍스트는 모델 경계뿐만 아니라 시스템의 물리적 경계 역할도 함

### 소유권 경계

바운디드 컨텍스트는 한 팀에서만 구현, 발전, 유지 관리 해야 함

## 실생활의 바운디드 컨텍스트

- 소프트웨어에서 비즈니스 도메인을 모델링할 때 다른 컨텍스트에서 다른 모델을 사용함

### 시맨틱 도메인

- 시맨틱 도메인은 의미 영역과 해당 의미를 전달하기 위해 사용하는 단어 영역으로 구분됨

### 과학

- 모든 경우에 맞는 과학적 이론은 없으며, 다른 이론은 다른 맥락 안에서 유용함

### 냉장고 구입

- 골판지 조각은 냉장고처럼 보이지 않지만, 어떤 모델의 크기를 선택할지 결정할 때 매우 유용함
